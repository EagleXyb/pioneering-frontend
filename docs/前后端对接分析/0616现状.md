## 二、数据库表结构要点

数据库已完整设计了 **单 Agent (ReAct 模式)** 的核心表：

- `chat_sessions` — 会话，含 `agent_mode` 字段（`react_agent` / `rag_agent` / `NULL`）
- `chat_messages` — 消息主干
  - `content` (TEXT) — 纯文本摘要
  - `content_blocks` (JSONB) — **核心字段**，存储 Agent 执行步骤轨迹 / 多模态内容块
  - `prompt_tokens`, `completion_tokens`, `latency_ms` — LLM 可观测性指标
  - `user_rating`, `user_feedback` — 用户反馈闭环
- `agent_tool_executions` — 工具执行明细
  - `tool_name`, `input_params` (JSONB), `output_result` (TEXT), `output_summary` (TEXT)
  - `status` (pending / running / success / error / timeout / canceled)
  - `start_time`, `end_time`, `duration_ms`（自动计算）
- `token_usage` — Token 用量记录，含 `cost` 字段
- `user_quotas` — 用户配额表（`total_tokens`, `used_tokens`, `daily_limit`, `daily_used`）
- `users` — 用户表（支持密码、微信 openid / unionid）
- `refresh_tokens` — Token 刷新表（支持多设备）
- `files` — 文件表

## 三、API 接口设计

### 3.1 通用规范

- **Base URL**: `https://api.yourdomain.com/v1`
- **鉴权**: JWT Bearer Token (`Authorization: Bearer <token>`)
- **通用错误响应**:

```json
{
  "code": 400,
  "message": "请求参数错误",
  "details": "prompt: 消息内容不能为空",
  "requestId": "req_xxx"
}
```

### 3.2 核心对话接口（AG-UI SSE 流式）

三个端点共享相同的请求格式和 SSE 协议，区别仅在于 `agent_state` 事件的发送频次和颗粒度。

#### 请求格式

```json
POST /api/agent/{chat|analyze|task}
Content-Type: application/json
Authorization: Bearer <token>

{
  "conversationId": "sess_xxx",
  "prompt": "分析销售流失原因",
  "model": "gpt-4o-mini",
  "temperature": 0.7,
  "maxTokens": 2000,
  "parentMessageId": null
}
```

**参数说明**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `conversationId` | string | 否 | 会话 ID，为空则后端自动创建新会话 |
| `prompt` | string | 是 | 用户消息内容 |
| `model` | string | 否 | 模型标识，默认值由会话配置决定 |
| `temperature` | number | 否 | 生成温度 0-2，默认 0.7 |
| `maxTokens` | integer | 否 | 最大输出 Token 数，默认 2000 |
| `parentMessageId` | string | 否 | 父消息 ID，用于重新生成/编辑/分支对话 |

#### SSE 事件格式

**Content-Type**: `text/event-stream`

##### 事件 1：`agent_state` — 驱动 ProcessPanel / TaskPipeline

```sse
event: agent_state
data: {"key":"step_1","type":"thinking","label":"分析用户需求","status":"running"}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | string | 步骤唯一标识，如 `step_1`, `tool_search` |
| `type` | `thinking` \| `tool_call` \| `observation` \| `answer` | 步骤类型 |
| `label` | string | 步骤标题，显示在前端面板上 |
| `content` | string | 步骤详细内容（可选），展开时展示 |
| `status` | `running` \| `done` \| `pending` | 状态 |

##### 事件 2：`message` — 驱动消息列表

```sse
event: message
data: {"role":"assistant","content":[{"type":"text","data":"分析结果..."}]}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `role` | `assistant` | 固定 |
| `content` | ContentBlock[] | 支持 `text` / `code` / `image` 等类型 |

##### 事件 3：`done` — 结束信号

```sse
event: done
data: {"id":"msg_xxx","conversationId":"sess_xxx","usage":{"promptTokens":150,"completionTokens":320,"totalTokens":470}}
```

##### 事件 4：`error` — 流式错误

```sse
event: error
data: {"code":500,"message":"LLM 服务超时","requestId":"req_xxx"}
```

#### 三种模式的 SSE 差异

| 对比项 | Chat | Pro | Task |
|-------|------|-----|------|
| `agent_state` 事件 | 少发或不发 | 大量发送，每个步骤都发 | 发送多步骤流水线 |
| `message` 事件 | 直接输出回答 | 输出分析报告 | 输出执行结果汇总 |
| 典型步骤数 | 1-3 步 | 5-15 步 | 3-10 步 |

## 四、关键数据流

### Pro 模式完整流程

```
用户输入 "分析销售流失原因"
  → POST /api/agent/analyze { conversationId, prompt }
  → 后端创建/复用 chat_sessions (agent_mode='react_agent')
  → 存储用户消息到 chat_messages (role='user')
  → ReAct 循环:
      ├─ SSE: agent_state { key:"think_1", type:"thinking", label:"分析需求", status:"running" }
      ├─ SSE: agent_state { key:"think_1", type:"thinking", label:"分析需求", status:"done" }
      ├─ SSE: agent_state { key:"tool_1", type:"tool_call", label:"查询销售数据", status:"running" }
      │    └─ 写入 agent_tool_executions (tool_name, input_params, output_result, status, duration_ms)
      ├─ SSE: agent_state { key:"tool_1", type:"tool_call", label:"查询销售数据", status:"done" }
      ├─ SSE: agent_state { key:"obs_1", type:"observation", label:"数据结果", status:"done" }
      ├─ SSE: agent_state { key:"answer", type:"answer", label:"生成分析报告", status:"running" }
      ├─ SSE: message { role:"assistant", content:[{type:"text", data:"..."}] }
      └─ SSE: done { id, conversationId, usage }
  → 存储 assistant 消息到 chat_messages (content_blocks = 完整步骤轨迹)
  → 更新 chat_sessions (message_count, last_message_id, updated_at)
  → 更新 token_usage, user_quotas
```

## 五、前端对接后端接口的 5 个关键问题

### 问题 1：会话生命周期未与后端同步

当前 `conversationStore` 是**纯本地状态**，刷新页面即丢失。

**解决方案**：
- 首次发送消息时，如果 `activeId` 为空，先调 `POST /api/chat/sessions` 创建后端会话
- 用后端返回的 `session.id` 替换前端的 `activeId`
- 会话列表从 `GET /api/chat/sessions` 加载

### 问题 2：AG-UI 协议需要后端严格匹配

`useChat` 和 `useAgentState` 依赖 AG-UI 协议的特定 SSE 事件格式。后端必须按标准输出 `agent_state`、`message`、`done`、`error` 事件，否则前端无法正常消费。

### 问题 3：消息结构映射

| 前端字段 | 后端存储 | 说明 |
|---------|---------|------|
| `msg.content[]` (ContentBlock[]) | `chat_messages.content_blocks` (JSONB) | 完整富文本/步骤轨迹 |
| `msg.content` 中 text 拼接 | `chat_messages.content` (TEXT) | 纯文本摘要，用于列表预览 |
| `msg.role` | `chat_messages.role` | system/user/assistant/tool |

### 问题 4：历史消息加载

当前消息列表没有历史消息加载能力。需要：
1. 进入会话时调 `GET /api/chat/sessions/{sessionId}/messages?limit=30`
2. 将历史消息注入 `useChat`
3. 滚动到顶部触发游标分页加载更多

### 问题 5：用户认证

当前前端代码没有认证逻辑。需要：
- 登录页
- Token 存储（localStorage）
- HTTP 拦截器自动附加 `Authorization` header
- 401 时自动用 refresh token 续期或跳转登录

## 八、对接优先级

```
第一优先级（堵塞开发）
├── HTTP 客户端封装 + Token 管理     — 所有接口的前提
├── POST /api/agent/{chat|analyze|task} SSE  — 三个模式的核心通信通道
└── 会话首次创建 + sessionId 回写     — 否则消息无法持久化

第二优先级（核心体验）
├── 会话列表从后端加载               — 刷新不丢失
├── 历史消息游标分页加载             — 重新进入可查看历史
└── Pro 模式 content_blocks → stateMap 重建 — 历史会话展示推理过程

第三优先级（完善功能）
├── 消息反馈（点赞/点踩）
├── 停止生成
├── TaskPipeline 对接工具执行数据
├── 编辑/重新生成
└── 配额查询
```
